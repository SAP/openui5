Card Translations

Overview

Translation files (i18n.properties files) are defined by using the "i18n" property of the "sap.app" namespace of the manifest. After that the keys can be accessed with the double bracket syntax. The properties with double bracket syntax will be replaced with the translated texts during manifest processing.

Another way to access the translations is by using the i18n model the card automatically creates (see examples below). This model will also includes a set of predefined translations which are common for all cards and can be used by everyone. To differentiate between the card internal translations and the user defined ones the developer must use the "CARD." prefix (see the list of predefined translation keys below).

Examples

Translating properties during manifest processing (where the user i18n files contain keys "TITLE" and "SUBTITLE"):

{
	"sap.app": {
		"id": "card.explorer.translation.card",
		"type": "card",
		"i18n": "i18n/i18n.properties",
		"applicationVersion": {
			"version": "1.0.0"
		}
	},
	"sap.card": {
		"header": {
			"title": "{{TITLE}}",
			"subTitle": "{{SUBTITLE}}",
			...
		},
		...
	}
}

Translations in binding syntaxes by using the i18n model of the card (where the user i18n files contain keys "EXPECTED_ITEMS" and "SUBTITLE"):

{
	"sap.app": {
		"id": "card.explorer.translation.card",
		"type": "card",
		"i18n": "i18n/i18n.properties",
		"applicationVersion": {
			"version": "1.0.0"
		}
	},
	"sap.card": {
		"header": {
			"data": {
				"json": {
					"numberOfItems": "17"
				}
			}
			"title": "{i18n>EXPECTED_ITEMS} {/numberOfItems}",
			"subTitle": "{i18n>SUBTITLE}",
			...
		},
		...
	}
}
